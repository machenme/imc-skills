# PRD Template — 各节结构与格式

## 1. PM 备注 (Critical)

引用块形式，置于 PRD 最顶部：

- 用户想法中缺失的核心商业逻辑或闭环漏洞
- 建议的补救方向
- 若想法已足够完整，注明"当前方案逻辑闭环完整，可直接进入设计阶段"

## 2. 项目概述

> **[产品名称]**：一句话核心价值

必须显式列出：
- **核心目标**：要解决什么问题
- **明确不做**：当前阶段排除的功能/场景（没有边界的 PRD 是愿望清单）

## 3. 目标用户

2-3 个典型用户画像，每个包含：
- **画像名称**
- **典型特征**（年龄、职业、场景）
- **核心痛点**

## 4. 功能矩阵与业务说明

### 功能总览

| 优先级 | 功能名称 | 描述 |
|--------|----------|------|
| P0 (Must Have) | ... | 没有这些功能产品无法跑通核心闭环 |
| P1 (Should Have) | ... | 核心功能的延伸，极大提升体验 |
| P2 (Nice to Have) | ... | 锦上添花 |

### 核心业务流程说明

针对每个 P0 和 P1 模块展开业务流程描述：
- **[P0 功能名称] 业务流程**：...
- **[P1 功能名称] 业务流程**：...

## 5. 用户故事 & 验收标准 (AC)

P0 和 P1 核心功能，标准格式：

```
**Story N: [故事标题]** (P0)
> As a [角色], I want to [行为], so that [价值].

**验收标准：**
- Given [前置条件], When [用户操作], Then [预期结果]
```

## 6. 非功能需求

- **性能**：首屏加载时间、响应延迟上限（必须量化数值）
- **安全**：数据加密方式、隐私保护策略
- **兼容性**：支持的端（Web/iOS/Android/小程序等）

## 约束

- 每个功能描述必须具体，避免"优化体验""提升效率"等模糊措辞
- 非功能需求必须给出量化数值指标（毫秒、百分比、具体版本号）
- ToC 产品考虑冷启动或用户留存策略；ToB 产品考虑多租户/权限模型
- 禁止使用"尽快""安全"等模糊词汇
